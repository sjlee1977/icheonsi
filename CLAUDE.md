## 아키텍처

> 프로젝트의 기술 스택과 전체 구조를 여기에 설명하세요.
> 예시: Next.js 14 (App Router) + Supabase + Tailwind CSS

### 주요 디렉토리

```
app/          # Next.js App Router 페이지 및 레이아웃
components/   # 재사용 가능한 UI 컴포넌트
lib/          # 공통 유틸리티, 헬퍼, 클라이언트 설정
api/          # API 라우트 핸들러
directives/   # AI 에이전트용 SOP 지침 파일
execution/    # 자동화 작업용 결정론적 Python 스크립트
.tmp/         # 임시 파일 (커밋 금지, 항상 재생성 가능)
```

### 스타일링 규칙

- 조건부 클래스 병합에는 `lib/utils.ts`의 `cn()` (clsx + tailwind-merge) 사용
- `tailwind.config.ts`에 정의된 프로젝트 디자인 토큰 시스템을 따를 것
- 모든 페이지와 컴포넌트에서 일관된 톤 앤 매너 유지 — 타이포그래피 굵기, 색상 팔레트, 간격, 전체적인 미적 감각이 애플리케이션 전반에 걸쳐 시각적으로 조화로워야 함

## 에이전트 지침

신뢰성을 극대화하기 위해 관심사를 분리하는 3계층 아키텍처로 운영합니다. LLM은 확률적이지만 대부분의 비즈니스 로직은 결정론적이며 일관성이 필요합니다. 이 시스템은 그 불일치를 해결합니다.

### 3계층 아키텍처

**Layer 1: 지시 계층 (무엇을 할지)**
- `directives/`에 Markdown으로 작성된 SOP(표준 운영 절차)
- 목표, 입력값, 사용할 도구/스크립트, 출력값, 엣지케이스 정의
- 중간 직급 직원에게 주는 것처럼 자연어로 작성된 지침

**Layer 2: 오케스트레이션 계층 (의사결정 — AI의 역할)**
- 지시문을 읽고, 실행 도구를 올바른 순서로 호출하고, 오류를 처리하고, 필요 시 명확화 요청, 학습 내용을 지시문에 반영
- 의도와 실행 사이의 연결고리 역할
- 예시: 웹 스크래핑을 직접 하지 말고 — `directives/scrape_website.md`를 읽고 입력/출력을 결정한 뒤 `execution/scrape_single_site.py`를 실행

**Layer 3: 실행 계층 (실제 작업)**
- `execution/`에 있는 결정론적 Python 스크립트
- 환경 변수 및 API 토큰은 `.env`에 저장
- API 호출, 데이터 처리, 파일 작업, 데이터베이스 연동 담당
- 신뢰성 있고, 테스트 가능하며, 빠르게 동작. 주석을 충분히 달 것.

**이 구조가 효과적인 이유:** 모든 것을 직접 처리하면 오류가 누적됩니다. 단계별 90% 정확도라도 5단계를 거치면 59%로 떨어집니다. 복잡한 작업을 결정론적 코드로 밀어넣으면 AI는 순수하게 의사결정에만 집중할 수 있습니다.

## 운영 원칙

**1. 기존 도구 먼저 확인**
새 스크립트를 작성하기 전에 지시문에 따라 `execution/`에 기존 도구가 있는지 확인하세요. 없을 때만 새로 작성합니다.

**2. 오류 발생 시 자가 복구 (Self-anneal)**
- 에러 메시지와 스택 트레이스를 꼼꼼히 읽을 것
- 스크립트를 수정하고 재테스트
- 예외: 수정에 유료 API 호출이나 크레딧이 필요한 경우, 먼저 사용자에게 확인
- 배운 내용(API 제한, 타이밍, 엣지케이스, 더 나은 접근법)을 지시문에 업데이트
- 예시: API 속도 제한에 걸림 → API 조사 → 배치 엔드포인트 발견 → 스크립트 재작성 → 테스트 → 지시문 업데이트

**3. 학습 내용을 지시문에 반영**
지시문은 살아있는 문서입니다. API 제약, 더 나은 접근법, 자주 발생하는 오류, 타이밍 기대치를 발견하면 지시문에 업데이트하세요. 명시적으로 지시받지 않는 한 지시문을 새로 만들거나 덮어쓰지 마세요. 지시문은 장기적인 지침서이므로 즉흥적으로 사용하고 버리지 말고 꾸준히 개선해야 합니다.

**4. 톤 앤 매너 유지**
모든 서브페이지와 컴포넌트는 메인 디자인과 일관된 외관을 유지해야 합니다. 타이포그래피(굵기와 크기), 색상 팔레트, 간격, 전체적인 미적 감각이 포함됩니다. 디자인 결정은 애플리케이션 전반의 시각적 조화를 우선시해야 합니다. 사용자가 명시적으로 예외나 다른 방향을 요청하면 사용자의 지침을 따르세요.

**5. 생성보다 결정론적 방식 우선**
작업을 스크립트로 처리할 수 있다면 스크립트를 사용하세요. LLM의 추론은 진정한 판단이 필요한 작업에만 활용하세요: 라우팅, 오류 해석, 모호성 해소, 지시문 업데이트.

**6. 가정하기 전에 확인**
지시문이 없거나 모호하거나, 되돌릴 수 없는 작업(데이터 삭제, 결제, 메시지 발송 등)이 필요한 경우 진행 전에 반드시 사용자에게 확인하세요.

## 자가 복구 루프

오류는 학습의 기회입니다. 무언가 깨졌을 때:

1. **진단** — 전체 오류와 트레이스를 읽기
2. **수정** — 스크립트 또는 접근 방식 수정
3. **테스트** — 수정이 end-to-end로 동작하는지 확인
4. **업데이트** — 관련 지시문에 수정 내용 반영
5. 시스템이 이전보다 강해짐

## 파일 구성

**결과물 vs 중간 파일:**
- **결과물** — 사용자가 접근할 수 있는 클라우드 기반 출력물 (Google Sheets, Slides, 호스팅된 URL 등)
- **중간 파일** — 처리 중에만 필요한 임시 파일

**디렉토리 규칙:**
- `.tmp/` — 모든 중간 파일. 커밋 금지. 항상 재생성 가능.
- `execution/` — Python 스크립트 (결정론적 도구)
- `directives/` — Markdown SOP (지침서)
- `.env` — 환경 변수 및 API 키 (커밋 금지)
- `credentials.json`, `token.json` — OAuth 인증 정보 (`.gitignore`에 포함)

**핵심 원칙:** 로컬 파일은 처리 목적으로만 사용합니다. 최종 결과물은 사용자가 접근할 수 있는 클라우드 서비스에 저장합니다. `.tmp/`의 모든 파일은 언제든지 삭제하고 재생성할 수 있습니다.

## Claude Code 활용 팁 (2025)

- 새 프로젝트 시작 시 `/init`을 사용해 변경 전에 저장소 구조를 파악하게 할 것
- `/memory`를 활용해 세션 간 프로젝트 핵심 정보를 유지할 것
- 전체 파일 재작성보다 부분 수정을 선호할 것 — 작은 diff가 검토하기 쉽고 회귀 버그 가능성이 낮음
- 대규모 코드베이스 작업 시 검색을 요청하기보다 관련 파일 경로를 명시적으로 제공할 것
- 여러 AI 코딩 에이전트를 사용하는 경우 `AGENTS.md` 또는 `GEMINI.md`를 이 파일의 미러로 유지할 것
- 지시문은 구체적일수록 효과적: 예상 입력값, 출력값, 예시 값을 포함할 것
